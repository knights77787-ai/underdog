# 프론트–백엔드 연동 완성 · 2일 계획표

> **목표:** 피드백 API 연동 + (선택) 설정 연동까지 마치고, 라이브/커스텀 소리 흐름을 끝까지 검증한다.

---

## 전제

- **이미 완료:** 라이브(WS join, ts_ms), 커스텀 소리(POST /custom-sounds, .wav)
- **남은 핵심:** 피드백 연동 (event_id 필요 → 백엔드 보강 후 프론트 연동)
- **선택:** 설정 GET/POST 연동, 간단한 설정 UI

---

## 1일차

### 오전 (약 3~4시간): 피드백용 event_id · 백엔드

| 순서 | 작업 | 내용 | 완료 |
|------|------|------|------|
| 1-1 | alert에 event_id 포함 | `_persist_alert`가 생성한 `event_id`를 반환받아, 브로드캐스트용 `entry`에 `event_id` 추가 후 `broadcast_to_session` | [ ] |
| 1-2 | 스펙 반영 | `Shared/ws_protocol.md`의 alert 메시지 예시에 `event_id` 필드 문서화 | [ ] |
| 1-3 | 동작 확인 | WS로 alert 수신 시 payload에 `event_id` 들어오는지 로그/테스트 | [ ] |

**참고:**  
- `_persist_alert`는 현재 `create_alert_event`만 호출. `create_alert_event`가 `event_id`를 반환하도록 수정한 뒤, handlers에서 `entry["event_id"] = event_id` 설정 후 브로드캐스트.

### 오후 (약 3~4시간): 피드백 · 프론트

| 순서 | 작업 | 내용 | 완료 |
|------|------|------|------|
| 2-1 | 최근 alert 보관 | 라이브 화면에서 alert 수신 시 `event_id` 포함해 “최근 알림 1건”(또는 N건) 저장 (피드백 대상 지정용) | [ ] |
| 2-2 | 맞아요/아니에요 연동 | `btnFeedbackYes` → POST /feedback `vote: "up"`, `btnFeedbackNo` → `vote: "down"`. body: `event_id`, `vote`, (선택) `comment` | [ ] |
| 2-3 | 에러/성공 처리 | 4xx/5xx 시 토스트로 안내, 200 시 “피드백 저장됨” 등 토스트 | [ ] |
| 2-4 | 대상 없음 처리 | 아직 alert를 한 번도 안 받은 상태에서 피드백 버튼 누르면 “대상 알림이 없습니다” 등 안내 | [ ] |

**1일차 마감 체크:**  
- [ ] 백엔드 alert에 `event_id` 포함  
- [ ] 라이브에서 “맞아요”/“아니에요” 클릭 시 POST /feedback 호출 및 화면 피드백
